# QF-020 — Golden Rules

**Document ID:** QF-020

**Version:** 1.0.0

**Status:** Immutable Principles

**Owner:** QIRA Leadership

**Last Updated:** 2026

---

# Purpose

This document defines the Golden Rules of QIRA.

Unlike other Foundation documents, these principles are considered constitutional.

Every engineer, AI Employee, consultant, designer, partner, contractor, and future contributor must follow these rules.

If another document conflicts with these rules,

**these rules always win.**

---

# The Constitution of QIRA

Every decision made inside QIRA should answer one question:

> **"Does this move QIRA closer to becoming the world's best AI Workforce Platform?"**

If the answer is no,

do not build it.

---

# Rule 1

## Customer Value Before Technology

Technology exists to solve business problems.

Never build technology because it is interesting.

Build technology because it creates measurable value.

Always optimize for:

- Customer Success
- Business Outcome
- Simplicity
- Long-term Value

Never optimize for:

- Hype
- Trends
- Complexity
- Engineering Ego

---

# Rule 2

## Build Platforms, Not Projects

Every project must improve the platform.

Never ask

> "How do we build this for this customer?"

Instead ask

> "How do we build this once for every future customer?"

---

# Rule 3

## AI Is A Colleague

AI is never treated as a feature.

AI is part of the workforce.

Every AI Employee has

Responsibilities

Permissions

Knowledge

Memory

Accountability

Human Oversight

---

# Rule 4

## Humans Make Final Decisions

AI assists.

Humans approve.

AI recommends.

Humans are accountable.

Critical business decisions always remain human responsibilities.

---

# Rule 5

## Documentation Is Product

Documentation is never optional.

Documentation is not written after development.

Documentation is developed with the product.

Every important decision must be documented.

---

# Rule 6

## Security Is Mandatory

Security is never postponed.

Security is never optional.

Security is everyone's responsibility.

Every release must improve or preserve platform security.

---

# Rule 7

## Simplicity Wins

The simplest architecture that solves the problem is preferred.

Avoid:

Overengineering

Premature optimization

Unnecessary abstraction

Complex workflows

Simple systems survive.

---

# Rule 8

## Reuse Before Reinventing

Before writing code ask

Does this already exist?

Can another module solve this?

Can another service solve this?

Can another package solve this?

Can configuration solve this?

Reuse first.

---

# Rule 9

## Every Project Improves QIRA

Every implementation contributes

Knowledge

Patterns

Templates

Architecture

Documentation

Automation

Prompt Improvements

Industry Knowledge

No project leaves QIRA unchanged.

---

# Rule 10

## Company Brain Is Sacred

Knowledge is the most valuable asset of QIRA.

Never lose knowledge.

Never duplicate knowledge.

Never hide knowledge.

Every lesson becomes reusable.

---

# Rule 11

## Data Belongs To Customers

Customer data is never owned by QIRA.

QIRA protects customer data.

Uses customer data responsibly.

Never shares customer data.

Never trains models without explicit authorization.

Trust is more valuable than revenue.

---

# Rule 12

## Think In Systems

Never optimize one feature.

Optimize the platform.

Every decision should improve

Architecture

Maintainability

Developer Experience

Customer Experience

AI Capability

Knowledge

---

# Rule 13

## Configuration Before Custom Code

Always prefer

Configuration

↓

Workflow

↓

Template

↓

Plugin

↓

Custom Development

Custom code is the last option.

---

# Rule 14

## Every Decision Must Be Explainable

Architecture should be explainable.

AI should be explainable.

Pricing should be explainable.

Recommendations should be explainable.

If we cannot explain it,

we should rethink it.

---

# Rule 15

## Never Break Trust

Never exaggerate AI capability.

Never fabricate facts.

Never hide limitations.

Never mislead customers.

Trust compounds.

Broken trust is difficult to recover.

---

# Rule 16

## Build For Ten Years

Every architectural decision should assume QIRA will still use it in ten years.

Shortcuts create future costs.

Think beyond today's customer.

---

# Rule 17

## Ship, Learn, Improve

Perfection is impossible.

Learning is mandatory.

Release.

Measure.

Learn.

Improve.

Repeat.

---

# Rule 18

## AI Should Continuously Learn

Every AI interaction should improve

Prompt Library

Knowledge

Company Brain

Patterns

Automation

Templates

AI should become better every day.

---

# Rule 19

## Every Engineer Is An Architect

Engineers do not simply implement.

They improve.

Question assumptions.

Suggest improvements.

Protect the platform.

Think long-term.

---

# Rule 20

## Every AI Is An Engineer

AI should

Understand context.

Review architecture.

Suggest improvements.

Generate documentation.

Write tests.

Explain decisions.

Think before coding.

---

# Rule 21

## Product Before Features

Users buy outcomes.

Not features.

Every feature must answer

What customer problem does this solve?

If no answer exists,

the feature should not exist.

---

# Rule 22

## Measure Everything

If it matters,

measure it.

Examples

Platform Usage

AI Usage

Customer Satisfaction

Knowledge Growth

Delivery Speed

Architecture Quality

Automation Success

Engineering Quality

What cannot be measured should still be observable.

---

# Rule 23

## One Source Of Truth

Every important artifact has exactly one authoritative source.

Examples

Foundation

↓

Architecture

↓

Database

↓

API

↓

Knowledge

↓

Documentation

Duplicate sources create inconsistency.

---

# Rule 24

## AI Must Respect Boundaries

AI never bypasses

Permissions

Security

Compliance

Privacy

Human Approval

AI capability never overrides governance.

---

# Rule 25

## Leave The Platform Better Than You Found It

Every Pull Request.

Every Project.

Every Discovery.

Every Customer.

Every AI Conversation.

Should improve something.

Even if only documentation.

Continuous improvement defines QIRA.

---

# Decision Checklist

Before approving any work ask:

Does it create customer value?

Does it strengthen the platform?

Can it be reused?

Is it secure?

Is it documented?

Does it improve the Company Brain?

Will future engineers thank us?

If the answer is mostly "No",

stop and redesign.

---

# The QIRA Oath

Every contributor commits:

> I will build with integrity.

> I will protect customer trust.

> I will document my decisions.

> I will improve the platform.

> I will think beyond today's implementation.

> I will leave QIRA better than I found it.
